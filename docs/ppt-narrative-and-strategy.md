# AI4Math 投资人 Pitch Deck — 叙事结构与战略反思

## Part I: PPT 叙事结构（共 18 页）

---

### 第 1 页 · 封面

**AI4Math — 形式化定理证明的 Agent 基础设施**

副标题: 不只是让 AI 更会做数学 —— 重新设计"AI 做数学"的工具本身

---

### 第 2 页 · 数学为什么重要

**数学是一切科学的基础语言**

- 物理、密码学、金融、芯片验证 —— 都建立在数学证明之上
- 全球数学研究年产 12 万+ 篇论文，但每篇论文的证明验证仍依赖人类同行评审
- 2024 年 Fields 奖获得者的一项工作花了 7 年才被社区确认无误
- 如果有一台机器能自动验证甚至发现数学证明，它改变的不只是数学本身

核心论点: 数学证明是知识确定性的最高标准，也是 AI 最有价值的前沿之一

---

### 第 3 页 · 形式化——让数学可被机器处理

**形式化证明 = 每一步推理都被计算机内核逐行验证**

- Lean4 / Coq / Isabelle 等形式化证明助手已运行 30+ 年
- Mathlib（Lean4 最大数学库）: 16 万+ 个已验证定理，100 万+ 行代码
- 里程碑: 2017 年 Kepler 猜想被形式化验证、2023 年 PFR 猜想被 Lean 社区在数月内形式化
- 但: 形式化一个已知定理平均需要 10-50 倍于非形式证明的人力

痛点: 形式化证明太慢、太痛苦、太依赖稀缺的人类专家

---

### 第 4 页 · 人类做证明——为什么痛苦

**人类证明的认知过程:**

1. 直觉形成 → 高度依赖数学家个人经验
2. 策略选择 → 只能串行尝试，赌一个方向
3. 局部推理 → 频繁需要"记住"大量上下文
4. 卡住回溯 → 认知成本极高，往往沉没数小时
5. 整理定稿 → 发现之前的"显然"其实有洞

本质瓶颈: 串行、单路径、回溯昂贵 → 高度依赖"选对方向"的天才直觉

---

### 第 5 页 · 机器辅助证明——十年演进

**2015-2022: 漫长的探索期**

| 年份 | 里程碑 | 方法 | 局限 |
|------|--------|------|------|
| 2016 | Holophrasm | 神经引导搜索 (Metamath) | 证明率仅 14% |
| 2019 | DeepHOL | RL + 交互环境 (HOL Light) | 动作空间受限 |
| 2020 | GPT-f | Transformer + 搜索 | Metamath 限定 |
| 2021 | TacticZero | 端到端深度 RL (HOL4) | 参数生成弱 |
| 2022 | Thor | LLM + Hammer 混合 | 依赖 ATP 可用性 |

核心困境: 稀疏奖励 + 数据匮乏 + 证明器接口笨重 → 进展缓慢

### 第 6 页 · Agent 时代——条件成熟

**2023-2025: 三个条件同时到位，进展加速**

条件一: 大模型涌现推理能力 → GPT-4 / Claude 可以生成合理的证明草稿
条件二: LeanDojo / PISA 等项目把证明器封装为可交互环境 → RL 和搜索有了"gym"
条件三: RL 从证明器反馈中学习（DeepSeek-Prover, AlphaProof）→ 自我提升闭环形成

关键转折: 2024 年 7 月，AlphaProof 在 IMO 上解出 3 道非几何题，达到银牌水平
→ 2025 年登上 Nature，标志着 AI 数学推理从实验室进入竞争力级别

---

### 第 7 页 · 技术主线已收敛

**所有前沿工作都收敛到同一框架:**

"把证明器当成可交互环境，用模型生成 tactic，用搜索/RL 探索状态空间"

四大范式（往往混合使用）:
1. Tactic 级搜索 Agent: MCTS / best-first + 模型生成 tactic（AlphaProof, DeepSeek-Prover）
2. 整体证明 + 修复: 一次生成 → 编译报错 → 修复（Baldur, APOLLO）
3. 检索增强: Mathlib 10 万+ 引理的精准匹配（LeanDojo/ReProver, COPRA, Rango）
4. 自举/自博弈: 自动出题 + 自动证明，对抗数据稀缺（STP, LEGO-Prover）

miniF2F 基准 SOTA: DeepSeek-Prover-V2 达到 88.9% (pass@8192)

---

### 第 8 页 · 但所有人都忽略了一个问题

**所有工作都在"证明器之上"做文章，没有人改造证明器本身。**

这就好比: 所有棋类 AI 团队都在改进搜索算法和神经网络，但都在用同一个 1990 年代设计的棋盘引擎。

Lean4 是为"一个人在编辑器前写证明"设计的:
- 状态不可分叉 → Agent 想同时尝试 1000 个 tactic，需要 1000 次 deepcopy
- 每步完整检查 → 搜索阶段 99% 的分支是死胡同，完整检查是浪费
- 返回人类可读的错误字符串 → Agent 需要结构化信息才能调整策略
- LSP 会话有内存泄漏 → 长时间运行的搜索会崩溃

这是基础设施层的瓶颈，不是算法层可以解决的。

---

### 第 9 页 · 我们的洞察

**Agent 的核心优势是"大规模并行树搜索 + 零成本回溯"**

人类: 串行探索，每步赌一个方向，回溯代价高
Agent: 可以同时尝试 1000 条路，走不通立刻丢弃，零沉没成本

但 Lean4 的架构把 Agent 的这个优势完全限制住了。

我们的目标: 设计一个从内核到协议层都为 Agent 优化的验证引擎，
释放 Agent 在并行搜索上的全部潜力。

---

### 第 10 页 · AI4Math 系统架构

**双引擎架构: Lean4 兼容 + APE (Agent Proof Engine)**

路径 A · Lean4 传统路径:
LLM 生成完整证明 → Lean4 编译器验证 → 与 Mathlib 生态完全兼容

路径 B · APE Agent-First 引擎:
持久化状态 → tactic 级并行搜索 → L0/L1 快速过滤 → L2 最终认证

两条路径互补: APE 做快速搜索探索，Lean4 做最终可信认证。

六大 Agent 模块: 思考大脑 · 知识检索 · 工具生态 · 形式验证 · 自演化 · 上下文记忆

---

### 第 11 页 · APE 引擎——五大创新

1. **持久化证明状态 (O(1) 分叉)**
   实测: 200 ns/fork vs deepcopy 的 10,845 μs = 54,000x

2. **三层验证 (10-1000x 搜索加速)**
   L0 (~1μs) 过滤 80% → L1 (~100μs) 过滤 15% → L2 仅验证最终证明

3. **显式约束图**
   Metavariable 耦合关系被追踪 → Agent 知道哪些子目标可安全并行

4. **结构化错误反馈**
   返回 `{kind: "arity_mismatch", expected: 3, got: 1}` 而非字符串

5. **Token 高效的状态视图**
   多粒度表示: 最小摘要(给LLM) / 结构化(给匹配) / 完整(给调试)

### 第 13 页 · 与竞品的关键差异

**我们不是在 Lean4 上做更好的搜索算法 —— 我们重新设计了搜索的基础设施**

| 维度 | AlphaProof / DeepSeek 等 | AI4Math + APE |
|------|--------------------------|---------------|
| 验证引擎 | 使用 Lean4 as-is | 自研 APE + Lean4 互补 |
| 状态分叉 | 受 Lean4 限制 (~ms) | O(1) 持久化 (~200ns) |
| 验证策略 | 每步完整检查 | 三层渐进 (L0/L1/L2) |
| 错误信号 | 解析 Lean4 错误文本 | 原生结构化反馈 |
| 可扩展性 | 受 LSP 会话限制 | 无状态 worker，水平扩展 |
| 开放性 | 多数闭源或部分开源 | 完全开源 |

类比: 他们在 CPU 上优化算法，我们在做 GPU 硬件。

### 第 14 页 · 关键科学问题与技术难点

**科学问题: 如何让验证引擎的设计与 Agent 搜索模式最优匹配？**

技术难点:
1. 持久化数据结构能否表达 CIC 类型理论的全部复杂性？(metavar 耦合问题)
2. 分层验证的 L1 层如何在速度和可靠性之间取得最优平衡？(false positive 率)
3. .olean 格式读取与 Mathlib 兼容的工程挑战
4. Python 原型到 Rust 生产引擎的迁移路径

我们的解决方案:
- metavar 耦合: 显式约束图 + are_independent() 判断
- L1 平衡: 可配置的 reduction depth + 来自搜索结果的在线校准
- 兼容性: 单向兼容策略 (读 .olean, 导出 proof term)
- 迁移: Phase 1 Python 验证 → Phase 2 Rust core + PyO3 绑定

## Part II: 战略反思与竞争分析

### 一、我们与同期工作相比的真正创新点

纵观调研报告中的 30+ 个公开工作，我们的项目有一个**独特的定位**：

**所有其他工作都在"验证器之上"做创新（更好的模型、更好的搜索、更好的数据），
只有我们在改造验证器本身。**

这是一个基础设施层的创新，而非算法层的创新。

具体来说:

1. **AlphaProof** (Google DeepMind): 在 Lean 环境中做 AlphaZero 式 RL + test-time RL。
   非常强大，但把 Lean 当作黑箱。如果 Lean 的 tactic state 管理更高效，
   AlphaProof 的搜索效率会显著提升。我们的 APE 引擎正好解决了这个问题。

2. **DeepSeek-Prover-V2**: 子目标分解 + RL。它的 MCTS 搜索受 Lean 交互延迟限制。
   论文本身提到了 RMaxTS 的内在奖励来减少无效探索——
   我们的分层验证 (L0/L1) 是另一种更直接的方式: 在源头过滤。

3. **LeanDojo**: 提供 Lean 的 Python 交互 API。它是"环境封装层"的工作，
   我们的 APE 是"替代环境层"——不是把 Lean 封装得更好用，
   而是提供一个为 Agent 从头设计的替代方案。

4. **LEGO-Prover / STP**: 通过自博弈和技能库增长来对抗稀疏奖励。
   这些是"学习层"的创新，与我们的"基础设施层"创新是正交且互补的。
   它们可以运行在 APE 上获得更高吞吐。

### 二、项目目前的不足（坦诚评估）

1. **类型检查器不完整**: 当前只实现了 CIC 的核心子集（Pi, Lambda, App, Const, Var, MVar）。
   完整的 Lean4 兼容需要: universe polymorphism, quotient types, proof irrelevance,
   mutual inductive definitions, type class resolution。
   差距: 大约完成了 30% 的类型理论实现。

2. **没有 .olean 读取器**: 号称可以导入 Mathlib，但实际上这个组件尚未实现。
   没有它，APE 就是一个"玩具环境"而非"可用工具"。

3. **没有真实基准数据**: 尚未在 miniF2F 或任何公开基准上测试。
   目前跑通的只是手工构造的简单定理 (P → P, modus ponens 等)。

4. **没有 RL/搜索集成**: APE 引擎提供了搜索树和批量 tactic API，
   但尚未接入任何 LLM tactic generator 或 RL 训练循环。

5. **"分层验证"的 L1 层过于简化**: 当前的 unification 和 WHNF reduction
   只覆盖了最基本的 case。真实的 Lean4 elaboration 包含大量启发式规则
   (type class synthesis, instance resolution, coercion)，这些在 L1 中被跳过，
   可能导致 false positive 率过高。

6. **Python 性能天花板**: 虽然持久化数据结构本身很快 (~200ns fork)，
   但实际的 tactic 执行（包含 unification）在 Python 中约 20-30μs。
   对于 100K+ 节点搜索，这可能不够。Rust 迁移是必须的，但尚未开始。

### 三、同期工作中我们应该学习的

1. **从 DeepSeek-Prover-V2 学**: 子目标分解的思想。当前 APE 的 tactic
   只有 intro/assumption/apply/sorry，没有"分解子目标"的能力。
   V2 的结果表明分解是处理复杂定理的关键。

2. **从 LEGO-Prover 学**: 可增长的技能库。APE 的 search tree 是 per-proof 的，
   没有跨问题的知识积累机制。引入 skill/lemma bank 是明显的增强方向。

3. **从 Rango 学**: 不只检索引理，还检索相似证明。这对 LLM 的 prompt 质量
   提升巨大，应尽早集成到 agent 的 brain 模块中。

4. **从 AlphaProof 学**: Test-time RL 的威力。在推理时对特定问题做在线训练，
   是解决真正困难问题的关键。APE 的低延迟状态管理天然适合这种模式。

5. **从 ProofAug 学**: 多粒度自动化增强。在 tactic 执行中自动调用
   内置 automation (simp, omega, decide 等) 可以极大减少搜索深度。

### 四、未来迭代方向的判断

**短期 (3 个月) 必须做的:**
- 完成 .olean 读取器 → 否则没有实际使用场景
- 在 miniF2F 上建立可复现基准 → 否则没有可信度
- 接入 LLM tactic generator → 否则只是一个空转的引擎

**中期 (6-12 个月) 的关键决策:**
- Rust 迁移是否用 PyO3 (保持 Python 接口) 还是全 Rust (更高性能但更高学习成本)
- 是否支持 Coq/Isabelle (扩大用户基础但分散精力)
- RL 训练循环的设计: online RL vs offline RL vs test-time RL

**长期 (12+ 个月) 的趋势判断:**

趋势 1: **Agent 证明将从"竞赛数学"走向"工程验证"**。
2026 年的 Agentic Proof Automation 案例研究表明，
Agent 在真实项目中的 proof engineering 效率已达 87%。
这意味着商业化的第一个落地场景可能不是"证明新定理"，
而是"自动化已知定理的形式化"。

趋势 2: **验证器生态将从"单一证明器"走向"多后端"**。
ProofGym 已在做统一 Python API 覆盖 Coq/Isabelle/Lean。
APE 如果能成为这个统一后端中"最快的 Agent 友好选项"，
就有机会成为标准基础设施。

趋势 3: **"搜索"和"学习"将更紧密耦合**。
AlphaProof 的 test-time RL 表明：推理时学习是解决真正困难问题的关键。
APE 的低延迟状态管理天然适合高频搜索-学习循环。
这可能是 APE 最大的长期价值——不是比 Lean4 快几倍，
而是让全新的搜索-学习范式成为可能。

### 五、对投资人要诚实说明的风险

1. **执行风险**: 完整的 CIC 类型检查器实现是一项重大工程挑战。
   如果做不好，APE 永远停在"玩具"阶段。

2. **生态风险**: Lean4 社区如果自己做了 Agent 友好的改造
   (如 Kimina Lean Server 的进一步发展)，APE 的独立价值会降低。

3. **市场风险**: 形式化证明目前仍是小众领域。
   商业化可能需要等待"Agent 证明"在芯片/合约验证中的采用加速。

4. **竞争风险**: Google (AlphaProof) 和 DeepSeek 有远超我们的算力和人才。
   我们的优势不在模型或数据，而在基础设施层的差异化。
