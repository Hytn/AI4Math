## 一、项目骨骼：一句话理解 AI4Math

**AI4Math 是什么？**

一个让数百个 AI 数学家像真实的研究院一样协同工作的操作系统。

它不是一个"更好的证明生成器"——它是一整套基础设施：底层提供毫秒级验证能力，中层积累可复用的数学知识，上层协调异构智能体群落分工合作。三层形成飞轮，越用越聪明。

**与当前 SOTA 的本质差异：**

```
当前范式 (DeepSeek/Goedel/Kimina):
    LLM → 生成完整证明 → Lean4 编译 (2-12s) → pass/fail (1 bit) × N 次独立重试
    ❌ 每轮只获得 1 bit 信息  ❌ 方向之间零通信  ❌ 失败经验完全丢失

AI4Math 范式:
    N 个异构智能体 → 验证 OS (目标 50ms) → 结构化反馈 (~100 bits) × 实时广播
    ✅ 丰富的错误诊断指导下一步  ✅ 发现即刻共享  ✅ 知识自动沉淀复用
```

**四根支柱，一个飞轮：**

```
┌─────────────────────────────────────────────────────────────────┐
│  ④ 数学家社会 — 异构智能体群落的自组织协作                         │
│     数百个角色各异的智能体，分工探索、实时通信、合力攻克难题          │
├─────────────────────────────────────────────────────────────────┤
│  ③ 世界模型 — 证明器的心智模拟器                                  │
│     内化 Lean4 状态转移动力学，不调用证明器即可预判策略效果           │
├─────────────────────────────────────────────────────────────────┤
│  ② 活知识系统 — 自演化的数学记忆                                  │
│     从海量证明经验中涌现层次化知识，具备遗忘、修正、重组能力          │
├─────────────────────────────────────────────────────────────────┤
│  ① 验证操作系统 — 无限弹性的算力基底                              │
│     毫秒级响应 · 弹性伸缩 · 增量编译 · 热缓存 · 持久化上下文        │
├─────────────────────────────────────────────────────────────────┤
│  ⓪ Lean4 REPL — 100% 精确的形式化验证器                          │
└─────────────────────────────────────────────────────────────────┘

                     ↑ 知识沉淀 ↓ 知识注入 — 飞轮自转
```

## 二、项目脊柱：一道定理的完整生命之旅

理解 AI4Math 最好的方式是追踪一道定理从输入到被证明的全过程。以下是数据如何在四层架构中流动的：

```
用户输入一道定理
       │
       ▼
  ┌─────────────┐     ┌──────────────────────────────────────┐
  │ 读题 & 分析  │────▶│ 领域识别 · 难度评估 · 策略规划          │
  └──────┬──────┘     └──────────────────────────────────────┘
         │
         ▼
  ┌─────────────┐     ┌──────────────────────────────────────┐
  │ 知识注入     │────▶│ 从知识金字塔提取:                      │
  │ (Layer ②)   │     │   相关引理 · tactic 建议 · 策略模式     │
  └──────┬──────┘     │   错误模式 (避免重蹈覆辙)               │
         │            └──────────────────────────────────────┘
         ▼
  ┌─────────────┐     ┌──────────────────────────────────────┐
  │ 多方向并行   │────▶│ N 个异构智能体同时探索:                 │
  │ (Layer ④)   │     │   · 直接证明 / 反证 / 归纳 / 分解...   │
  └──────┬──────┘     │   · 各有独立模型、温度、角色             │
         │            └──────────────────────────────────────┘
         ▼
  ┌─────────────┐     ┌──────────────────────────────────────┐
  │ 三级验证     │────▶│ L0: 语法预过滤 (~1μs)  — 瞬间淘汰废案  │
  │ (Layer ①)   │     │ L1: REPL 快验 (~50ms)  — 结构化反馈    │
  └──────┬──────┘     │ L2: 全编译   (~3s)     — 终极确认      │
         │            └──────────────────────────────────────┘
         ▼
  ┌─────────────┐     ┌──────────────────────────────────────┐
  │ 实时广播     │────▶│ 成功 → 全局共享已证引理                 │
  │ (Layer ④)   │     │ 失败 → 广播负面知识 (剪枝)              │
  └──────┬──────┘     │ 部分 → fork 给其他方向继续              │
         │            └──────────────────────────────────────┘
         ▼
  ┌─────────────┐     ┌──────────────────────────────────────┐
  │ 知识沉淀     │────▶│ 自动写入知识金字塔:                     │
  │ (Layer ②)   │     │   L0: 原始证明轨迹 (世界模型训练数据)    │
  └─────────────┘     │   L1: tactic 效能统计 · 错误模式        │
                      │   L2: 归纳出的策略模式                  │
                      │   L3: 跨领域概念关联图谱                │
                      └──────────────────────────────────────┘
         │
         ▼
    下一道题自动继承所有积累的知识 → 飞轮加速
```

**关键洞察：** 知识不是附加功能，而是贯穿整个管道的血液。每一次证明尝试（无论成败）都在让系统变得更聪明。

## 三、项目关节：模块如何彼此连接

理解了骨骼（是什么）和脊柱（数据怎么流），现在看各模块之间的接口——它们如何"铆接"在一起。

### 3.1 验证操作系统 (`engine/`, 11,779 行)

智能体的"物理世界"——所有证明尝试最终都要经过这里。

```
                    engine/ 内部结构
┌──────────────────────────────────────────────┐
│  对外接口: AsyncOrchestrator                  │
│  "给我一个定理，我返回证明或结构化失败原因"      │
├──────────────┬───────────────────────────────┤
│ 调度层       │ verification_scheduler.py      │
│              │ resource_scheduler.py          │
│              │ → 三级验证分流 + 资源调度        │
├──────────────┼───────────────────────────────┤
│ 池化层       │ async_lean_pool.py             │
│              │ pool_scaler.py                 │
│              │ → N 路 REPL 长连接 + 弹性伸缩   │
├──────────────┼───────────────────────────────┤
│ 加速层       │ incremental_verifier.py        │
│              │ prefilter.py                   │
│              │ → 增量编译 + 语法预过滤          │
├──────────────┼───────────────────────────────┤
│ 持久化层     │ proof_context_store.py         │
│              │ proof_session.py               │
│              │ → 跨会话状态 + 训练数据导出      │
├──────────────┼───────────────────────────────┤
│ 通信层       │ broadcast.py                   │
│              │ transport.py                   │
│              │ → 发布订阅总线 + 协议抽象        │
├──────────────┼───────────────────────────────┤
│ 诊断层       │ error_intelligence.py          │
│              │ observability.py               │
│              │ → 结构化反馈 + 全链路可观测      │
└──────────────┴───────────────────────────────┘
```

**关键接口：**
- `AsyncLeanPool.try_tactic(env_id, tactic)` → 向任意 REPL 发送 tactic，返回结构化结果
- `VerificationScheduler.verify(candidate)` → 自动走三级验证流水线
- `BroadcastBus.publish(channel, message)` → 跨智能体实时通信

### 3.2 活知识系统 (`knowledge/`, 2,193 行)

从证明经验中自发涌现层次化知识的活系统。

```
        知识金字塔 (SQLite + WAL)
┌────────────────────────────────────┐
│  L3: 直觉图谱  concept_nodes/edges │ ← 跨领域概念关联
├────────────────────────────────────┤
│  L2: 策略模式  strategy_patterns   │ ← 归纳出的可迁移策略
├────────────────────────────────────┤
│  L1: 战术知识  tactic/lemma/error  │ ← 具体效能统计
├────────────────────────────────────┤
│  L0: 原始轨迹  traces/trajectories │ ← 完整状态转移 (训练数据)
└────────────────────────────────────┘

  写入路径:  验证结果 → KnowledgeBroadcaster → KnowledgeWriter → SQLite
  读取路径:  KnowledgeReader.render_for_prompt() → 注入 LLM prompt
  生命周期:  KnowledgeEvolver.decay_tick() 衰减 · gc_stale() 清理 · revive() 恢复
```

**与 engine 的铆接：**
- `AsyncOrchestrator._producer` 调用 `KnowledgeReader` 注入知识到 prompt
- `AsyncOrchestrator._verify_candidate` 调用 `KnowledgeBroadcaster` 沉淀结果
- `ProofContextStore.export_rich_trajectories()` 导出世界模型训练数据

### 3.3 多智能体协作 (`agent/` + `prover/pipeline/`)

异构智能体群落的调度与协同。

```
  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐
  │Generator│  │ Planner │  │ Repair  │  │ Critic  │  ... (11 种角色)
  └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘
       │            │            │            │
       └────────────┴────────────┴────────────┘
                         │
                    BroadcastBus          ← 发现即刻共享
                         │
               ┌─────────┴──────────┐
               │  AsyncOrchestrator  │    ← 生产-消费流水线
               │  Producer → Queue → Consumer
               └─────────┬──────────┘
                         │
                    engine/ (验证)
                    knowledge/ (沉淀)
```

**关键设计：**
- **角色适配：** 11 种角色 × 3 级模型 (Haiku/Sonnet/Opus)，按任务复杂度自动匹配
- **钩子系统：** 9 个生命周期时机（`on_before_llm`, `on_tactic_result`, `on_proof_found` ...）
- **插件生态：** YAML 声明式策略配置，按领域加载（数论、代数、分析）

### 3.4 世界模型 (训练数据管道已就绪)

```
  每次 tactic 执行自动记录:
    (proof_state, action, next_state, error, goal_diff)
                    │
                    ▼
    ProofContextStore.export_rich_trajectories()
                    │
                    ▼
    RichProofTrajectory — 可直接用于训练 transformer-based 世界模型
                    │
                    ▼
    目标: 智能体在心智空间中模拟策略效果，无需调用 Lean4
```

## 常见问题

**Q: 和 DeepSeek-Prover 等系统的根本区别是什么？**
它们是"更强的证明生成器"，AI4Math 是"让证明生成器在其中运行的操作系统"。就像 Linux 之于应用程序——AI4Math 提供调度、内存、通信、存储等基础设施，任何 LLM 都可以作为上层的证明生成器插入。

**Q: 能支持 Coq / Isabelle 吗？**
REPL 交互通过 `Transport(ABC)` 抽象，知识系统和智能体层不含 Lean4 特定代码。扩展到其他证明器只需新增 Transport 实现。

**Q: 世界模型的训练数据从哪来？**
`ProofContextStore.export_rich_trajectories()` 导出完整的 `(state, action, next_state, error)` 转移记录，可直接用于训练。
