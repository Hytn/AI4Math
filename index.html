为什么做这个项目

当前范式的根本瓶颈

所有 SOTA 系统（DeepSeek-Prover、Goedel-Prover、Kimina-Prover）本质上都是同一个范式：让 LLM 生成完整证明，然后扔给 Lean4 编译。三个根本问题无人解决。

当前 SOTA 范式

LLM 生成完整证明 → Lean4 完整编译（2-12秒）→ pass/fail（1 bit 信息）→ 独立重试 N 次

❌ 每轮只获得 1 bit 反馈 ❌ N 个并行方向之间零通信 ❌ 失败尝试的知识完全丢失
AI4MATH 范式

N 个异构智能体 → 验证 OS（目标 50ms）→ 结构化反馈（~100 bits）→ 实时广播共享

✅ 结构化错误诊断指导下一步 ✅ 一个方向的发现即刻惠及所有方向 ✅ 知识自动沉淀，越用越聪明

四根支柱

不是更好的证明器，是证明器的操作系统

AI4Math 构建四层基础设施，形成自增强的飞轮。任何 LLM 都可以作为上层的证明引擎插入这个平台。

常见问题

FAQ

和 DeepSeek-Prover 等系统的根本区别是什么？
它们是"更强的证明生成器"，AI4Math 是"让证明生成器在其中运行的操作系统"。就像 Linux 之于应用程序——AI4Math 提供调度、内存、通信、存储等基础设施，任何 LLM 都可以作为上层的证明引擎插入。
能支持 Coq / Isabelle 吗？
REPL 交互通过 Transport(ABC) 抽象，知识系统和智能体层不含 Lean4 特定代码。扩展到其他证明器只需新增 Transport 实现。
知识系统使用什么存储？
SQLite + WAL 模式，单机支持百万级条目并发读写。分布式场景可替换为 PostgreSQL，接口不变。
世界模型的训练数据从哪来？
ProofContextStore.export_rich_trajectories() 导出完整的 (state, action, next_state, error) 转移记录。每次 tactic 执行自动采集，零额外标注成本。
无需 API Key 也能运行吗？
可以。bash eval.sh 默认使用 Mock 模式，走通完整管线包括知识读写和广播通信。
